解析ELF可执行文件

发表于 2025-08-13 更新于 2025-09-01 分类于 嵌入式
本文通过 readelf 命令深入解析 ELF 格式可执行文件的内部结构，详细讲解代码段、数据段、符号表等关键段的功能与意义，帮助读者理解二进制文件的组成原理。

当我们编译 C/C++ 程序生成可执行文件后，双击运行即可得到结果，但很少有人深究这个二进制文件内部到底长什么样。实际上，可执行文件并非一堆杂乱无章的 0 和 1，而是按照严格格式组织的结构化数据。在 Linux 系统中，最常见的可执行文件格式是 ELF（Executable and Linkable Format），而readelf命令则是剖析 ELF 文件的利器。本文将通过readelf的各种参数，逐层揭开可执行文件的神秘面纱，带你看清它的 “内部器官”—— 各个段（Section）的真实面貌。

二、ELF 文件的整体结构概览

首先用readelf -h查看 ELF 文件的头部信息，了解文件的基本属性：

输出结果（精简版）：

头部信息揭示了关键信息：

  • Class: 64 位 ELF 文件（对应 x86_64 架构）

  • Type: 可执行文件（EXEC）

  • Entry point address: 程序入口地址（0x400430，即main函数附近的代码起始位置）

  • Section headers: 段表信息（共 31 个段，段表起始位置在文件偏移 6240 字节）

ELF 文件的整体结构可分为三部分：

  1. ELF 头部：如上述信息，描述文件的基本属性

  2. 段表（Section Headers）：记录所有段的元信息（名称、类型、大小、偏移量等）

  3. 段数据：各个段的实际内容（代码、数据、符号表等）

输出结果中会列出 31 个段（因编译器版本可能略有差异），每个段有名称、类型、地址、偏移量、大小等属性。其中最关键的段包括：

  • 代码相关：.text（代码段）、.plt（过程链接表）

  • 数据相关：.data（已初始化数据段）、.bss（未初始化数据段）、.rodata（只读数据段）

  • 符号相关：.symtab（符号表）、.strtab（字符串表）、.dynsym（动态符号表）

三、代码段：程序的 “行动指令”

代码段（.text）是可执行文件中最核心的段之一，存放程序的机器指令（二进制形式的代码）。通过readelf -x .text可查看其十六进制内容：

.text段的特点：

  • 只读：防止程序运行时意外修改指令（内存保护机制）

  • 可执行：CPU 可直接从该段读取并执行指令

  • 位置无关（部分情况下）：动态链接时可加载到任意内存地址

四、数据段：程序的 “记忆仓库”

数据段用于存放程序中的变量，根据初始化状态分为.data、.bss和.rodata三个关键段。

1. .data：已初始化的全局 / 静态变量

我们在程序中定义了int global_var = 10;（已初始化全局变量），其值存放在.data段。通过readelf -x .data查看：

输出结果：

  • 0x00601028是global_var的内存地址

  • 0a000000是十六进制的10（小端存储，低字节在前）

  • 后续的00000000可能是其他已初始化变量的占位符

2. .bss：未初始化的全局 / 静态变量

未初始化的全局变量（如global_uninit_var）或初始值为 0 的变量存放在.bss段。与.data不同，.bss段在文件中不占用实际空间（仅记录大小），程序加载时由操作系统分配内存并初始化为 0。

通过readelf -S查看.bss段的信息：

  • NOBITS：表示该段无实际数据（仅占内存，不占文件空间）

  • Size为000008（8 字节）：global_uninit_var是 int 类型（4 字节），可能还有其他未初始化变量

3. .rodata：只读数据

字符串常量（如”Hello, ELF!”）和const修饰的全局变量存放在.rodata段，该段只读不可修改。

输出结果：

  • 48656c6c6f2c20454c4621对应 ASCII 码的”Hello, ELF!”

  • 末尾的00是字符串结束符\0

.rodata段的只读特性由内存页保护机制保证，试图修改会触发段错误（Segmentation fault）。

五、符号表：程序的 “身份档案”

符号表（.symtab）记录了程序中所有符号（变量名、函数名）的信息，包括符号类型、地址、大小等。通过readelf -s查看：

输出结果（精简版）：

各字段含义：

  • Value：符号的内存地址（如global_var在0x601028）

  • Size：符号大小（global_var是 int 类型，占 4 字节）

  • Type：符号类型（OBJECT表示变量，FUNC表示函数）

  • Ndx：所在段的索引（25对应.bss段，14对应.text段）

符号表的作用：

  • 编译链接阶段：确保函数和变量的引用正确解析

  • 调试阶段：gdb通过符号表将内存地址映射为变量名 / 函数名

  • 动态链接：动态加载器通过符号表定位共享库中的函数

六、其他重要段：程序的 “辅助器官”

除了核心的代码段和数据段，ELF 文件还有多个辅助段，负责程序的加载、链接和调试。

1. .plt与.got：动态链接的 “桥梁”

当程序使用动态链接库（如printf来自libc）时，.plt（Procedure Linkage Table，过程链接表）和.got（Global Offset Table，全局偏移表）负责在运行时找到共享库函数的实际地址。

  • .plt：存放跳转指令，首次调用共享库函数时触发动态链接器解析地址

  • .got：存放已解析的共享库函数地址，后续调用直接使用，避免重复解析

通过readelf -x .plt可查看.plt段的内容，其中包含指向.got的跳转指令。

2. .shstrtab与.strtab：字符串表

  • .shstrtab（Section Header String Table）：存放段表中所有段的名称（如.text、.data）

  • .strtab（String Table）：存放符号表中符号的名称（如函数名、变量名）

通过readelf -p .shstrtab demo可查看段名称字符串：

输出结果中包含所有段的名称，如[ 11] .text、[ 17] .rodata等。

七、程序头表：告诉操作系统如何加载

程序头表（Program Headers）描述了 ELF 文件如何被操作系统加载到内存，通过readelf -l查看：

输出结果中的 “Segment”（段，与 Section 不同）表示内存加载单元，例如：

  • LOAD类型的 Segment：将文件中的部分内容加载到内存，包含.text、.data等 Section

  • Dynamic类型的 Segment：存放动态链接信息

每个 Segment 有VirtAddr（内存地址）、PhysAddr（物理地址）、FileSiz（文件中大小）、MemSiz（内存中大小）等属性，操作系统根据这些信息分配内存并加载数据。

八、总结：可执行文件的 “解剖图”

通过readelf的各种命令，我们可以清晰看到 ELF 可执行文件的内部结构：

段名称 功能 关键特性
.text 存放机器指令 只读、可执行
.data 存放已初始化全局 / 静态变量 可读写、占文件空间
.bss 存放未初始化全局 / 静态变量 可读写、不占文件空间
.rodata 存放只读数据（字符串常量等） 只读
.symtab 记录符号信息（变量 / 函数名） 调试和链接的基础
.plt/.got 动态链接时定位共享库函数 实现延迟绑定

这些段按照 ELF 格式的规则有序组织，共同构成了可执行文件的 “躯体”。操作系统加载程序时，根据程序头表将各段映射到内存的相应位置，CPU 从.text段的入口地址开始执行指令，通过符号表和动态链接机制调用外部函数，通过数据段读写变量 —— 整个过程如同一个精密的机器在运转。

理解可执行文件的结构，不仅能帮助我们排查链接错误、优化程序性能，更能深入理解程序从源码到运行的完整生命周期。readelf作为剖析工具，为我们打开了一扇窥探二进制世界的窗口，让看似神秘的可执行文件变得清晰可见。

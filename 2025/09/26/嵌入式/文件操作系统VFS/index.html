虚拟文件系统VFS

发表于 2025-09-26 更新于 2025-10-10 分类于 嵌入式
本文字数： 2.3k 阅读时长 ≈ 3 分钟
本文介绍虚拟文件系统VFS。

我们从零基础开始，一步一步来理解 虚拟文件系统（VFS，Virtual File System）。

一、什么是文件系统？

在讲 VFS 之前，先搞清楚 文件系统（File System） 是什么。

  • 文件系统 是操作系统用来组织、存储和管理磁盘（或其他存储设备）上文件的一种机制。
  • 不同的操作系统、不同的存储设备，可能使用不同的文件系统。例如：
    • Windows 常用：NTFS、FAT32
    • Linux 常用：ext4、XFS、Btrfs
    • macOS 常用：APFS、HFS+

每种文件系统都有自己的规则：怎么存文件名、怎么记录文件大小、怎么分配磁盘空间等等。

二、问题来了：操作系统怎么同时支持这么多文件系统？

假设你用的是 Linux，你可能同时挂载了：

  • 一个 ext4 格式的硬盘（Linux 原生）
  • 一个 FAT32 的 U 盘（来自 Windows）
  • 一个 NTFS 的移动硬盘
  • 甚至一个网络文件系统（比如 NFS）

如果每个程序都要知道“这个文件在 ext4 上怎么读，那个在 FAT32 上怎么写”，那太麻烦了！

于是，操作系统引入了一个“中间层”——这就是虚拟文件系统（VFS）。

三、VFS 是什么？——核心思想

VFS 是一个抽象层，它为上层应用程序和内核提供统一的文件操作接口，屏蔽底层不同文件系统的差异。

你可以把它想象成一个“翻译官”或“通用插座”：

  • 应用程序说：“我要打开 /home/user/file.txt”
  • VFS 说：“好的，我来处理。这个路径在哪个文件系统上？哦，在 ext4 分区上。”
  • 然后 VFS 调用 ext4 文件系统的具体实现去真正打开文件。
  • 如果路径是在 U 盘上（FAT32），VFS 就调用 FAT32 的实现。

对应用程序来说，完全感觉不到底层差异！ 它只需要用统一的系统调用（如 open(), read(), write()）就行。

四、VFS 的关键组成部分（简化版）

VFS 通过几个核心的“抽象对象”来统一管理不同文件系统：

1. 超级块（Superblock）

  • 描述一个已挂载的文件系统的整体信息。
  • 比如：文件系统类型（ext4？FAT32？）、总大小、可用空间、挂载点等。
  • 每个挂载的文件系统在 VFS 中都有一个对应的超级块。

2. 索引节点（Inode）

  • 描述一个文件或目录的元数据（不是文件名！）。
  • 包括：文件大小、权限、所有者、时间戳、数据块位置等。
  • 注意：文件名并不存在 inode 里，而是存在目录项中。

3. 目录项（Dentry，Directory Entry）

  • 表示路径中的一个组件，比如 /home/user/file.txt 中的 home、user、file.txt。
  • Dentry 把文件名和对应的 inode 关联起来。
  • VFS 会缓存 dentry，加速路径查找（比如你反复访问同一个文件）。

4. 文件对象（File）

  • 表示一个打开的文件实例。
  • 包含：当前读写位置（文件偏移）、访问模式（只读/读写）、指向 inode 的指针等。
  • 每次你 open() 一个文件，内核就创建一个 file 对象。

五、举个例子：打开一个文件的过程

假设你在 Linux 中执行：

cat /mnt/usb/data.txt

而 /mnt/usb 是一个 FAT32 格式的 U 盘。

VFS 的工作流程如下：

  1. 解析路径：VFS 从根目录开始，逐级查找 /mnt → /mnt/usb → /mnt/usb/data.txt。
  2. 查找 dentry：每一步都查找或创建 dentry，最终找到 data.txt 对应的 dentry。
  3. 获取 inode：通过 dentry 找到该文件的 inode（FAT32 文件系统会提供自己的 inode 表示）。
  4. 创建 file 对象：调用 open() 系统调用，VFS 创建一个 file 对象。
  5. 调用具体文件系统的操作：VFS 知道这个 inode 属于 FAT32 文件系统，于是调用 FAT32 的 read 函数来真正读取数据。
  6. 返回数据给用户程序：cat 命令拿到数据并显示。

整个过程，cat 完全不知道底层是 FAT32 还是 ext4！

六、为什么叫“虚拟”文件系统？

  • 因为 VFS 本身不存储任何真实数据，也不直接管理磁盘。
  • 它只是一个抽象框架，定义了“所有文件系统都应该实现哪些操作”。
  • 真正干活的是具体的文件系统（ext4、FAT32 等），它们向 VFS “注册”自己的一套操作函数（比如 .open, .read, .write）。

这就像“接口”和“实现类”的关系（如果你学过面向对象编程）。

七、VFS 支持的不仅仅是磁盘文件系统！

VFS 的设计非常通用，所以 Linux 中很多“看起来像文件”的东西也通过 VFS 实现：

  • proc 文件系统（/proc）：展示进程信息（如 /proc/1234/status）
  • sysfs（/sys）：展示内核和设备信息
  • tmpfs：基于内存的临时文件系统
  • pipe、socket：某些情况下也被抽象为文件

这些都不是传统意义上的“磁盘文件”，但通过 VFS，你可以用 read()/write() 像操作普通文件一样操作它们！

八、总结：VFS 的作用

作用 说明
统一接口 应用程序无需关心底层文件系统类型
抽象隔离 将通用操作与具体实现分离
提高兼容性 轻松支持多种文件系统共存
提升性能 通过 dentry/inode 缓存加速路径查找
扩展性强 新文件系统只需实现 VFS 规定的接口即可接入

九、类比帮助理解

想象你去不同国家旅行：

  • 在中国吃饭用筷子
  • 在美国吃饭用刀叉
  • 在印度可能用手

但如果你有一个“万能餐具适配器”（VFS），你只需要说“我要吃饭”，它自动给你配上当地正确的餐具。

你（应用程序）不用学每种餐具怎么用，适配器（VFS）帮你搞定！

如果你是初学者，现在只需要记住：

VFS 是 Linux 内核中一个“通用文件操作层”，让所有程序都能用同样的方式访问各种不同的文件系统。

等你以后学操作系统、Linux 内核或做系统编程时，会更深入理解它的实现细节。

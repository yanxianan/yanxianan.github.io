994. 腐烂的橘子

题目：
你在一个 m x n 的网格中，每个格子代表一个橘子，状态可以是：

  • 0 表示空
  • 1 表示新鲜橘子
  • 2 表示腐烂橘子

每分钟，任何与腐烂橘子相邻（上下左右）的新鲜橘子都会被感染并变腐烂。

返回直到所有橘子都腐烂所需的最小分钟数。如果不可能让所有橘子都腐烂，则返回 -1。

示例 1：
输入：grid = [[2,1,1],[1,1,0],[0,1,1]]
输出：4

示例 2：
输入：grid = [[2,1,1],[0,1,1],[1,0,1]]
输出：-1
解释：左下角的橘子无法被感染。

示例 3：
输入：grid = [[0,2]]
输出：0
解释：没有新鲜橘子。

提示：

  • 1 <= grid.length <= 10
  • 1 <= grid[i].length <= 10
  • grid[i][j] 为 0, 1, 或 2

这道题是 广度优先搜索 (BFS) 在网格图（Grid Graph）中的 教科书级应用。

它是力扣（LeetCode）第 994 题。

如果说“岛屿数量”教你如何遍历（DFS/BFS皆可），那么“腐烂的橘子”则强制要求你使用 BFS，并且是 “多源 BFS” (Multi-source BFS)。

为什么？因为腐烂的过程是 “传染”。

所有烂橘子是 同时 向四周扩散的，而不是一个传完再传下一个。这就像病毒爆发，我们要算的是“第几分钟”全军覆没，这本质上是在求 最短路径/最小层数。

1. 明确问题描述

题目名称：腐烂的橘子 (Rotting Oranges)

题目大意：

在给定的 m x n 网格中，每个单元格有三种状态：

  • 0：空单元格。
  • 1：新鲜橘子。
  • 2：腐烂的橘子。

规则：

每一分钟，腐烂的橘子会使其 上下左右 4 个方向上相邻的新鲜橘子变成腐烂的。

请你计算直到单元格中 没有新鲜橘子 为止所必须经过的最小分钟数。

如果不可能让所有新鲜橘子都腐烂，请返回 -1。

示例：

  • 第 0 分钟：左上角有一个烂橘子。
  • 第 1 分钟：烂橘子的邻居（右边和下边）烂了。
  • 第 2 分钟：新烂的橘子的邻居烂了。
  • …
  • 第 4 分钟：全部烂完。返回 4。

2. 核心思路：为什么必须是 BFS？

2.1 为什么 DFS 不行？

DFS (深度优先) 是“一条道走到黑”。

假设左上角有一个烂橘子，DFS 会先把一条路上的橘子全部烂完，算出比如需要 10 分钟。

但实际上，可能右下角也有一个烂橘子，它烂到中间只需要 2 分钟。

DFS 无法模拟 “齐头并进” 的物理过程。

2.2 多源 BFS (Multi-source BFS)

通常的 BFS 是从一个起点开始扩散。

这道题可能有 多个 烂橘子（多个起点）。

做法很简单：

  1. 初始化：把 所有 初始状态为 2 的烂橘子，全部加入队列。
  2. 开始扩散：它们在队列里就像在同一个起跑线上，第一轮循环会处理所有这些起点的邻居。

3. 算法流程 (层序遍历)

我们需要两个关键变量：

  1. queue：存放当前这一轮所有烂橘子的坐标。
  2. freshCount：记录还剩多少新鲜橘子。
    • 作用：用来判断最后是否成功烂完，以及控制循环（如果没有新鲜橘子了，就没必要继续扩散了）。

步骤：

  1. 扫描全图：
    • 遇到 2：入队。
    • 遇到 1：freshCount++。
  2. BFS 循环：
    • 只要 queue 不空 且 freshCount > 0：
    • time++（时间过了一分钟）。
    • 批处理：记录当前队列大小 size（这一分钟内有多少个传染源）。
    • 取出这 size 个烂橘子，分别去感染上下左右：
      • 如果是 1：把它变成 2，freshCount--，入队（成为下一分钟的传染源）。
  3. 结果判定：
    • 如果 freshCount == 0：返回 time。
    • 如果 freshCount > 0：说明有些橘子被墙隔开了，永远烂不到，返回 -1。

5. 复杂度分析

时间复杂度：$O(M \times N)$

  • 网格中的每个格子最多被访问一次（入队一次，出队一次）。
  • 即使我们有两层循环（初始化 + BFS），总体操作次数是线性的。

空间复杂度：$O(M \times N)$

  • 队列的大小。在最坏情况下（比如整个网格全是烂橘子），队列需要存储所有格子的坐标。

6. 初学者常见疑问 (Q&A)

  • Q: 为什么循环条件里要加 freshCount > 0？
    • A: 这是一个防止“多算一分钟”的技巧。
    • 假设最后一轮，烂橘子把所有剩下的新鲜橘子都感染了。此时 freshCount 变为 0。
    • 如果不加这个判断，新感染的橘子已经进队了，循环会再跑一轮（队列不空）。虽然这新一轮不会感染任何东西，但 minutes 会再加 1，导致结果偏大。
    • 加上 freshCount > 0，一旦全感染完，立刻停止计时。
  • Q: 需要用 visited 数组吗？
    • A: 不需要。
    • 我们可以直接修改原数组 grid。
    • 把 1 改成 2，就天然起到了“标记已访问”的作用。再遇到 2 或者 0 我们都跳过，只找 1。
  • Q: 只有 0 和 2，没有 1 的情况会怎样？
    • A:
      • 初始化时 freshCount 为 0。
      • 代码一开始的 if (freshCount == 0) return 0; 会直接拦截，返回正确答案 0。

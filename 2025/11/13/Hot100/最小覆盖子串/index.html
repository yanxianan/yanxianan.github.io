76. 最小覆盖子串

题目：
给你一个字符串 s、一个字符串 t。返回 s 中涵盖 t 所有字符的最小子串。如果 s 中不存在涵盖 t 所有字符的子串，则返回空字符串 ""。

注意：

  • 对于 t 中重复字符，我们寻找的子字符串中该字符数量必须不少于 t 中该字符数量。
  • 如果 s 中存在这样的子串，我们保证它是唯一的答案。

这是力扣（LeetCode）第 76 题，也是 滑动窗口（Sliding Window） 系列中公认的 BOSS 级 题目。

之所以称它为困难（Hard），是因为它结合了 “不定长窗口” 的伸缩策略和 “哈希表计数” 的匹配逻辑，不仅要求你找出来，还要求找 “最小” 的，这对逻辑严密性要求极高。

1. 明确问题描述

题目名称：最小覆盖子串 (Minimum Window Substring)

题目大意：

给你两个字符串 s（源字符串）和 t（目标字符串）。请你在 s 中找出包含 t 中所有字符的 最短子串。

关键约束：

  1. 包含所有：如果 t 中包含重复字符（如 “AA”），那么结果子串中也必须至少包含两个 ‘A’。
  2. 最短：在所有满足条件的子串中，返回长度最小的那个。
  3. 不存在：如果找不到，返回空字符串 ""。
  4. 复杂度：要求 $O(n)$ 时间复杂度。

示例：

输入：s = "ADOBECODEBANC", t = "ABC"

输出："BANC"

解释：

  • “ADOBEC” 包含 “ABC”，长度 6。
  • “BECODEBA” 包含 “ABC”，长度 8。
  • “BANC” 包含 “ABC”，长度 4。
  • 最短的是 “BANC”。

2. 核心思路分析

2.1 为什么暴力法不可行？

暴力法是枚举 s 的所有子串，然后检查每个子串是否包含 t。

  • 子串数量级是 $O(n^2)$。
  • 检查一次需要 $O(m)$。
  • 总复杂度 $O(n^3)$ 或 $O(n^2)$。对于 $10^5$ 长度的字符串，绝对超时。

2.2 核心策略：滑动窗口 + “欠债还钱”模型

我们可以把 t 中的字符看作我们的 “债务”（Need）。

我们要控制一个窗口 [left, right] 在 s 上滑动：

  1. 进货（Expand）：right 向右扩张，遇到的字符加入窗口。如果是我们要找的字符，就说明还了一笔“债”。
  2. 收缩（Shrink）：当所有债务都还清（窗口内包含了 t 所有字符）时，说明当前窗口是一个可行解。
    • 这时候我们要尝试优化：left 指针向右移，尝试扔掉左边的字符。
    • 为什么要扔？ 因为我们要找最小的窗口。如果扔掉左边的字符后，窗口依然满足条件（比如扔掉了一个无关紧要的 ‘X’，或者多余的 ‘A’），那我们就得到了一个更短的答案。
    • 何时停止收缩？ 直到扔掉某个关键字符，导致窗口不再满足条件（“债”又没还清了），这时候停止收缩，继续让 right 向右找货。

2.3 如何判断“债务还清”？

我们需要两个哈希表（或数组）：

  • need：记录 t 中每个字符需要多少个。
  • window：记录当前窗口中，每个字符已经收集了多少个。

关键变量 valid：

为了避免每次都遍历哈希表去检查是否满足条件（那样检查一次是 $O(52)$ 或 $O(128)$），我们维护一个变量 valid。

  • 表示 “已经有多少种字符的数量达标了”。
  • 比如 t = "AAB", need = {'A':2, 'B':1}。
    • 当窗口里有两个 ‘A’ 时，valid 加 1（说明 A 这一类达标了）。
    • 当窗口里有一个 ‘B’ 时，valid 再加 1。
    • 当 valid == need.size() 时，说明所有种类的字符都齐了。

4. 复杂度分析

时间复杂度：$O(N + M)$

  • $N$ 是 s 的长度，$M$ 是 t 的长度。
  • 我们先遍历 t 初始化哈希表：$O(M)$。
  • 双指针核心：right 指针从 0 走到 $N$，left 指针也最多从 0 走到 $N$。每个字符最多“进窗”一次，“出窗”一次。
  • 哈希表操作是 $O(1)$。
  • 总复杂度是线性的。

空间复杂度：$O(|\Sigma|)$

  • 我们需要存储字符集大小的哈希表。对于 ASCII 字符，大小最多 128。
  • 可以认为空间复杂度是 $O(1)$（常数级）。

5. 深度解惑：最容易写错的细节

Q1: 为什么 if (window[c] == need[c]) 这里用 == 而不是 >=？

  • A: 这是一个精妙的去重逻辑。
  • 假设 need['A'] = 2。
  • 当窗口有 1 个 ‘A’，window['A'] 变 2：满足条件了，valid++。
  • 当窗口有 2 个 ‘A’，又来 1 个 ‘A’，window['A'] 变 3：虽然数量更多了，但“达标”这个状态没有变，所以不需要再次 valid++。我们只在“刚刚好满足”的那一瞬间记录。

Q2: 为什么 window[d] == need[d] 时要 valid--？

  • A: 对称逻辑。
  • 假设 need['A'] = 2，当前 window['A'] = 3。
  • 移出一个 ‘A’，剩 2 个：依然满足 >=2，valid 不用变。
  • 再移出一个 ‘A’，剩 1 个：坏了，不够了！ 此时状态从“达标”变成了“不达标”，所以 valid--。

Q3: 为什么结果要存 start 和 len，而不是直接存字符串？

  • A: 性能优化。
  • 如果在循环里写 res = s.substr(left, right - left)，每次更新结果都会进行字符串拷贝，这会把时间复杂度退化成 $O(N^2)$。
  • 只存下标，最后截取一次，保证 $O(N)$。

234. 回文链表

题目：
给你一个单链表的头节点 head，判断这个链表是否为回文链表。

示例 1：
输入：head = [1,2,2,1]
输出：true

示例 2：
输入：head = [1,2]
输出：false

提示：

  • 链表中节点的数目在范围 [1, 10⁵] 内
  • 0 <= Node.val <= 9

这道题是链表问题的集大成者。

它之所以经典，是因为它一道题考了三个知识点：

  1. 快慢指针找链表中点。
  2. 反转链表（刚才学过的）。
  3. 双指针逐个对比。

如果你能流畅解决这道题，说明你对链表的操作已经具备了“组合拳”的能力。

1. 明确问题描述

题目名称：回文链表 (Palindrome Linked List)

题目大意：

给你一个单链表的头节点 head，请你判断该链表是否为回文链表。

  • 回文：正着读和倒着读一样。例如 1 -> 2 -> 2 -> 1 或 1 -> 2 -> 3 -> 2 -> 1。

关键约束：

  • 时间复杂度：$O(n)$。
  • 空间复杂度：$O(1)$。
  • 注：如果允许 $O(n)$ 空间，直接把链表值复制到数组里用双指针做很简单，但面试通常要求 $O(1)$ 空间。

2. 核心思路分析

2.1 为什么不能直接从后往前读？

链表是单向的，没有 prev 指针。我们没法从尾巴倒着走到头。

2.2 核心策略：折叠对比法 (Find Middle & Reverse)

既然不能倒着走，我们能不能把链表的后半段“掰弯”，让它反过来指？

操作流程：

  1. 找中点：使用 快慢指针。快指针一次走两步，慢指针一次走一步。当快指针走到尽头时，慢指针刚好走到中间。
  2. 反转后半段：从慢指针的后面开始，将后半部分链表进行 原地反转（利用刚才学的反转链表技巧）。
  3. 对比：
    • 一个指针 p1 从头开始。
    • 一个指针 p2 从反转后的后半段头部开始。
    • 两个指针同时向后走，逐个比较数值。如果不相等，就不是回文。
  4. 恢复现场（可选但推荐）：为了保持原始数据的完整性，对比完后最好把后半段再反转回去。

[图解逻辑]

原始：1 -> 2 -> 3 -> 2 -> 1

中点：3

反转后半段：1 -> 2 -> 3 <- 2 <- 1 (注意：3 指向了 NULL，新的尾巴是 3，新的头是最后的 1)

对比路径：

  • 左边：1 -> 2 -> 3

  • 右边：1 -> 2 -> 3

    完全一致 $\rightarrow$ True。

4. 复杂度分析

时间复杂度：$O(n)$

  • 找中点：快指针遍历一次，约 $n/2$ 次操作。
  • 反转后半段：遍历后半段，约 $n/2$ 次操作。
  • 对比：遍历后半段，约 $n/2$ 次操作。
  • 恢复现场：约 $n/2$ 次操作。
  • 总操作次数是线性的，即 $O(n)$。

空间复杂度：$O(1)$

  • 我们利用了链表自身的指针进行反转，只申请了几个临时指针变量。

5. 初学者常见疑问 (Q&A)

  • Q: 奇数长度和偶数长度的处理有区别吗？
    • A: 在找中点时，fast->next != nullptr && fast->next->next != nullptr 这个条件保证了：
      • 奇数长度 (1-2-3-2-1)：slow 停在 3。后半段从 2 开始。
      • 偶数长度 (1-2-2-1)：slow 停在第一个 2。后半段从第二个 2 开始。
    • 对比时，我们只看 p2（后半段）是否走完。因为前半段可能比后半段多一个节点（奇数时中间那个），但这不影响回文判断，多出来的那个中间节点不需要对比。
  • Q: 为什么要恢复链表？
    • A: 在实际工程中，如果一个函数叫 isCheck...，它应该只是“读取”数据并返回判断结果，不应该“修改”原本的数据结构。如果你把传入的链表结构打乱了，可能会导致调用者的后续程序崩溃。这是一个很好的编程习惯。

15. 三数之和

题目：
给你一个整数数组 nums，判断是否存在三元组 [nums[i], nums[j], nums[k]] 满足：

  • i != j、i != k、j != k
  • nums[i] + nums[j] + nums[k] == 0

请你返回所有和为 0 且不重复的三元组。

这就是我刚才提到的“数组算法守门员”——三数之和 (3Sum)。

如果说“两数之和”是幼儿园，“移动零”是小学，“盛最多水的容器”是中学，那么这道“三数之和”就是大学入学考试。

它之所以难，不是因为算法复杂，而是因为它对**边界情况（Corner Cases）和去重（De-duplication）**的处理要求非常高。很多初学者写出的代码要么超时，要么输出了一堆重复的结果。

1. 明确问题描述

题目名称：三数之和 (3Sum)

题目大意：

给你一个整数数组 nums，判断是否存在三个元素 $a, b, c$ ，使得 $a + b + c = 0$。请你找出所有满足条件且不重复的三元组。

关键约束：

  • 不重复：结果集里不能包含相同的组合。例如 [-1, 0, 1] 和 [0, 1, -1] 被视为同一个结果，只能保留一份。
  • 时间复杂度：暴力法是 $O(n^3)$，这里我们需要优化到 $O(n^2)$。

示例：

输入：nums = [-1, 0, 1, 2, -1, -4]

输出：[[-1, -1, 2], [-1, 0, 1]]

2. 核心思路分析

2.1 降维打击：化繁为简

我们要找 $a + b + c = 0$。

如果我们固定其中一个数（假设是 $a$），那么问题就变成了：

在剩下的数中，找到两个数 $b$ 和 $c$，使得 $b + c = -a$。

是不是很眼熟？没错！这就变成了一个 “两数之和” 的问题。

既然我们需要遍历每一个数作为 $a$，如果对于每个 $a$ 都跑一遍 $O(n)$ 的两数之和算法，总复杂度就是 $O(n^2)$。这符合预期。

2.2 为什么必须先排序？

对于“两数之和”，我们用的是哈希表。但在“三数之和”里，哈希表很难处理“不重复”这个要求。

（比如 [-1, -1, 2]，哈希表很难判断这次的 -1 和上次的 -1 组成的结果是不是重复的）。

排序的威力：

  1. 去重容易：相同的数字排在了一起。如果我们刚才处理了 nums[i]，发现 nums[i+1] 和它一样，那直接跳过 nums[i+1] 就行了，因为它能找到的组合，nums[i] 肯定都已经找过了。
  2. 双指针更高效：有序数组找两数之和，不需要哈希表，直接用左右指针（类似“盛最多水的容器”的逻辑）夹逼即可。

2.3 算法流程 (双指针法)

  1. 排序：将数组从小到大排序。
  2. 遍历：用指针 i 遍历数组，nums[i] 就是我们固定的第一个数 $a$。
  3. 双指针：令 left = i + 1，right = n - 1。
    • 计算 sum = nums[i] + nums[left] + nums[right]。
    • 如果 sum == 0：找到啦！ 记录答案。然后 left 右移，right 左移（继续找这组 $a$ 下的其他可能）。
    • 如果 sum < 0：太小了，left 右移（让数变大）。
    • 如果 sum > 0：太大了，right 左移（让数变小）。
  4. 去重（最核心的细节）：
    • 针对 i 的去重：如果 nums[i] == nums[i-1]，说明这个基准数刚才用过了，直接 continue。
    • 针对 left 和 right 的去重：找到一组答案后，如果 nums[left] 和下一位一样，继续右移；right 同理。

4. 复杂度分析

时间复杂度：$O(n^2)$

  • 排序：$O(n \log n)$。
  • 双层循环：
    • 外层循环遍历 i，次数为 $n$。
    • 内层 while 循环双指针从两端向中间走，复杂度是 $O(n)$。
    • 嵌套起来是 $n \times n = O(n^2)$。
  • 总复杂度是 $O(n^2) + O(n \log n)$，舍去低阶项，最终为 $O(n^2)$。这是该题目前已知的最优解法。

空间复杂度：$O(1)$ (不考虑输出占用的空间)

  • 除了存储结果的 result 数组外，我们只使用了常数个变量 (i, left, right, sum)。
  • 注：C++ 标准库的 sort 实现（通常是 Introsort）可能会用到 $O(\log n)$ 的栈空间。

5. 初学者避坑指南 (Debug 常见错误)

  1. 忘记排序：不排序，双指针逻辑完全失效，去重也无法完成。
  2. i 的去重写错了位置：
    • 错误写法：if (nums[i] == nums[i+1]) continue;
    • 后果：会漏掉结果。比如 [-1, -1, 2]，如果跳过了第一个 -1，这组解就找不到了。
    • 正确写法：if (i > 0 && nums[i] == nums[i-1])。意思是“我只要第一个出现的 -1，后面重复出现的 -1 我都不要”。
  3. 找到答案后忘记去重并移动指针：
    • 很多同学写了 result.push_back(...) 后直接写 left++。
    • 如果有 [-2, 0, 0, 2, 2]，找到 [-2, 0, 2] 后，如果不跳过中间的重复 0，下一次循环还会再找到一次 [-2, 0, 2]，导致结果重复。

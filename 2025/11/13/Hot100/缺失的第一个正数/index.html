41. 缺失的第一个正数

题目：
给你一个未排序的整数数组 nums，请你找出其中最小的缺失正整数。

示例 1：
输入：nums = [1,2,0]
输出：3
解释：正整数 1 和 2 存在，但 3 缺失。

示例 2：
输入：nums = [3,4,-1,1]
输出：2
解释：正整数 1 存在，2 缺失。

示例 3：
输入：nums = [7,8,9,11,12]
输出：1
解释：没有出现 1，因此最小缺失正整数是 1。

提示：

  • 1 <= nums.length <= 5 * 10⁵
  • -5 * 10⁵ <= nums[i] <= 5 * 10⁵

这道题是力扣（LeetCode）第 41 题，标记为 Hard。

很多同学一看到 “Hard” 就想退缩，但实际上这道题的难点不在于算法逻辑有多复杂，而在于它对空间复杂度的苛刻要求。

它要求我们在 $O(n)$ 时间内解决问题，同时只能使用 $O(1)$ 的额外空间。这意味着我们不能创建新的哈希表，也不能使用递归（栈空间），只能在原数组上动脑筋。

这道题是学习 “原地哈希 (In-place Hashing)” 技巧的最佳教具。

1. 明确问题描述

题目名称：缺失的第一个正数 (First Missing Positive)

题目大意：

给你一个未排序的整数数组 nums。请你找出其中没有出现的最小的正整数。

关键约束：

  • 时间复杂度：$O(n)$。
  • 空间复杂度：$O(1)$。

示例：

  1. 输入：[1, 2, 0] $\rightarrow$ 输出：3 (1和2有了，下一个是3)
  2. 输入：[3, 4, -1, 1] $\rightarrow$ 输出：2 (有1，缺2)
  3. 输入：[7, 8, 9, 11, 12] $\rightarrow$ 输出：1 (最小的正整数1就不在)

2. 核心思路分析

2.1 缩小搜索范围 (关键洞察)

假设数组长度为 $N$。

我们需要找的“缺失的第一个正数”，它的值会在什么范围内？

  • 最好的情况：数组里刚好是 [1, 2, 3, ..., N]。此时缺失的是 $N+1$。
  • 其他情况：数组里有一些负数、或者很大的数（比如 100）、或者重复的数。这些都会占据位置，导致 [1, N] 之间的某个整数无法出现。
  • 结论：答案一定在 $[1, N+1]$ 之间！

这意味着，我们只需要关注 $1$ 到 $N$ 这几个数字是否存在。对于负数、0、或者大于 $N$ 的数，我们根本不用管，它们是“干扰项”。

2.2 核心策略：萝卜坑原理 (原地哈希)

既然我们只关心 $1$ 到 $N$，我们可以把原数组当成一个哈希表来用。

规则设计：

我们希望把数字 x 放到它应该出现的位置上。

  • 数字 1 应该放在下标 0 的位置。
  • 数字 2 应该放在下标 1 的位置。
  • …
  • 数字 k 应该放在下标 k-1 的位置。

算法流程：

  1. 归位 (Swapping)：

    遍历数组，当我们遇到一个数字 x (即 nums[i]) 时，如果它满足以下三个条件，我们就把它交换到它正确的位置 (nums[x-1]) 去：

    • 它是正数 (x > 0)。
    • 它在有效范围内 (x <= N)。
    • 它当前不在正确的位置上，且目标位置上没有放正确的数 (避免死循环，比如 [1, 1])。

  2. 查岗 (Scanning)：

    归位操作完成后，我们再遍历一遍数组。

    • 检查下标 0，是不是 1？是则通过。
    • 检查下标 1，是不是 2？是则通过。
    • …
    • 如果在下标 i 处发现 nums[i] != i + 1，说明 $i+1$ 就是我们要找的那个缺失的数！

  3. 兜底：

    如果所有位置都对上了（说明数组是 [1, 2, ..., N]），那么缺失的就是 $N+1$。

4. 复杂度分析

时间复杂度：$O(n)$

很多同学看到 for 循环里套了个 while，第一反应是 $O(n^2)$。

这是错误的。实际上是 $O(n)$。

  • 分析：每次 swap 操作，都会把一个数字放到它最终正确的位置上。
  • 一旦一个数字归位了（比如 3 到了下标 2），它以后就不会再被移动了。
  • 对于长度为 $n$ 的数组，最多只有 $n$ 个位置，所以最多只能发生 $n$ 次有效的交换。
  • 均摊下来，时间复杂度是线性的。

空间复杂度：$O(1)$

  • 我们直接在输入数组 nums 上进行交换修改，只使用了常数个额外变量。
  • 这也就是所谓的“原地算法”。

5. 初学者常见疑问 (Q&A)

  • Q: 为什么要用 while 而不是 if？
    • A: 这是一个极好的细节。
    • 举例：nums = [3, 1, ...]，i=0。
    • 如果是 if：遇到 3，把它换到下标 2。此时下标 0 换回来了一个 1。如果不继续处理，指针 i 就往后走了，这个 1 就被留在了下标 0（虽然它恰好在正确位置，但如果是其他数就会错）。
    • 如果是 while：遇到 3，换走。换回来 1。while 继续检查这个 1，发现它应该去下标 0，正好在，循环结束。或者换回来一个 5，继续把 5 换到下标 4… 直到换回来一个没法处理的数（负数或已归位的数）。
  • Q: nums[nums[i] - 1] != nums[i] 这个条件是干嘛的？
    • A: 防止死循环和处理重复元素。
    • 假设数组是 [1, 1]。
    • 当处理第二个 1 时，它想去下标 0。但下标 0 已经有个 1 了。
    • 如果没有这个判断，程序会无限交换这两个 1，导致超时 (TLE)。这个条件保证了“如果目标位置已经是正确的数了，我就不折腾了”。

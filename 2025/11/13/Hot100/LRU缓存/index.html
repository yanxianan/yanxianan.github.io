146. LRU 缓存

题目：
设计并实现一个 LRUCache 类：

  • LRUCache(int capacity) 以正整数作为容量 capacity 初始化 LRU 缓存。
  • int get(int key) 如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1。
  • void put(int key, int value) 如果关键字已经存在，则变更其数据值；如果不存在，则插入该组「关键字-值」。当缓存达到其容量时，它应该在写入新数据之前删除最久未使用的数据值。

示例：
输入：
[“LRUCache”, “put”, “put”, “get”, “put”, “get”, “put”, “get”, “get”, “get”]
[[2], [1, 1], [2, 2], [1], [3, 3], [2], [4, 4], [1], [3], [4]]
输出：
[null, null, null, 1, null, -1, null, -1, 3, 4]
解释：
LRUCache lRUCache = new LRUCache(2);
lRUCache.put(1, 1); // 缓存是 {1=1}
lRUCache.put(2, 2); // 缓存是 {1=1, 2=2}
lRUCache.get(1); // 返回 1
lRUCache.put(3, 3); // 该操作会使得关键字 2 作废，缓存是 {1=1, 3=3}
lRUCache.get(2); // 返回 -1 (未找到)
lRUCache.put(4, 4); // 该操作会使得关键字 1 作废，缓存是 {4=4, 3=3}
lRUCache.get(1); // 返回 -1 (未找到)
lRUCache.get(3); // 返回 3
lRUCache.get(4); // 返回 4

提示：

  • 1 <= capacity <= 3000
  • 0 <= key <= 10⁴
  • 0 <= value <= 10⁵
  • 最多调用 get 和 put 方法 2 * 10⁵ 次

这道题是力扣（LeetCode）第 146 题，难度 Medium（但其实是各大厂面试中的 Hard 级常客，字节跳动甚至有一段时间是“必考题”）。

它的难点不在于算法有多深奥，而在于它要求你在 $O(1)$ 的时间内完成数据的 查询 (Get) 和 写入 (Put)，并且在容量满时自动淘汰 最近最少使用 的数据。

单一的数据结构（数组、链表、哈希表）都无法同时满足这些要求。你需要把 哈希表 和 双向链表 结合起来，设计一个复合数据结构。

1. 明确问题描述

题目名称：LRU 缓存 (LRU Cache)

题目大意：

设计一个满足 LRU (最近最少使用) 缓存约束 的数据结构。

实现 LRUCache 类：

  • LRUCache(int capacity): 以正整数作为容量 capacity 初始化 LRU 缓存。
  • int get(int key): 如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1。
  • void put(int key, int value):
    • 如果关键字 key 已经存在，则变更其数据值 value。
    • 如果不存在，则向缓存中插入该组 key-value。
    • 如果插入操作导致关键字数量超过 capacity，则应该 逐出 最久未使用的关键字。
  • 进阶要求：get 和 put 必须以 $O(1)$ 平均时间复杂度运行。

示例：

容量 = 2。

put(2, 2) -> [1, 2] (2 是最新的)

get(1) -> 返回 1。[2, 1] (1 刚被用过，变成最新的，2 变老了)

2. 核心思路分析

2.1 为什么单一结构不行？

  • 只用哈希表：查询快 $O(1)$，但不知道谁最老（没顺序）。
  • 只用链表：能维持顺序（新的放头，老的在尾），但查询慢 $O(N)$。

2.2 核心设计：哈希链表 (Hash Linked List)

我们需要把它们结合起来：

  1. 双向链表 (Doubly Linked List)：用来维护数据的时间顺序。
    • 规定：链表头部存最近使用的，链表尾部存最久未使用的。
    • 当数据被访问（Get）或更新（Put）时，把它移到链表头部。
    • 当容量满时，直接删除链表尾部的节点。
    • 为什么是双向？ 因为我们需要在 $O(1)$ 时间内删除任意一个节点，单向链表删除需要遍历找前驱，是 $O(N)$。
  2. 哈希表 (Hash Map)：用来实现 $O(1)$ 的查询。
    • Key: 题目给的 key。
    • Value: 指向链表中对应 节点 (Node) 的指针。
    • 这样我们就能直接通过 Key 找到链表里的节点，不需要遍历链表。

  • HashMap: {1: NodeA, 2: NodeB}
  • LinkedList: Head <-> NodeA(1,1) <-> NodeB(2,2) <-> Tail

4. 复杂度分析

时间复杂度：$O(1)$

  • get: 哈希查找是 $O(1)$，链表移动节点是 $O(1)$。
  • put: 哈希插入/删除是 $O(1)$，链表插入/删除节点是 $O(1)$。
  • 完美符合题目要求。

空间复杂度：$O(capacity)$

  • 我们需要存储最多 capacity 个节点，哈希表和链表都需要这么多空间。

5. 初学者常见疑问 (Q&A)

  • Q: 为什么 Node 里既要存 value 也要存 key？只存 value 不行吗？
    • A: 这是一个非常关键的细节！
    • 当缓存满时，我们删除了链表尾部的节点 node。
    • 此时，我们也必须从哈希表 cache 中把对应的记录删掉。
    • 可是哈希表的 API 是 cache.erase(key)。如果我们手里的 node 只有 value，我们怎么知道它的 key 是多少？
    • 所以节点里必须存 key，主要是为了反向查找哈希表并删除。
  • Q: 为什么可以用 std::list 吗？
    • A: 可以。C++ 的 std::list 就是双向链表。
    • 用 std::list 代码会短很多，但是面试官通常希望看你手写双向链表的操作（考察基本功）。
    • 另外，std::list 的迭代器操作稍微复杂一点，手写 Node 结构反而逻辑更直观。

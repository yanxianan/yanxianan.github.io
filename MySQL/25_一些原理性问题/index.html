MySQL一些原理性问题

发表于 2026-06-16 分类于 MySQL
本文字数： 6.4k 阅读时长 ≈ 9 分钟
总结 MySQL 使用中的一些常见原理性问题。

索引：从原理到实战

B+ 树原理

B+ 树是 MySQL InnoDB 存储引擎索引的底层数据结构。理解它，先对比一下几种树：

结构 特点 为什么不适合做数据库索引
二叉搜索树 每个节点2个子节点 数据量大时树太高，磁盘IO太多
平衡二叉树（AVL） 高度平衡，但仍是二叉 同上，树高 = log₂N，100万数据需要20层
B 树 多路平衡搜索树，节点存数据 非叶子节点也存数据，每层能存的键更少
B+ 树 多路平衡，只有叶子节点存数据，叶子节点之间有链表相连 ✅ 完美适合磁盘存储

B+ 树的核心优势：

  • 矮胖结构：每个节点可以有几百甚至上千个子节点（取决于页大小），100万数据可能只需要3层
  • 叶子节点链表相连：范围查询（如 WHERE id BETWEEN 100 AND 200）只需在叶子层顺序遍历，不用回溯
  • 非叶子节点只存键：不存数据，所以每层能容纳更多键，树更矮

💡 InnoDB 的页大小默认 16KB，一个 INT 键占 4 字节 + 指针 6 字节 ≈ 10 字节，一个非叶子节点页可存约 1600 个键。三层 B+ 树可存：1600 × 1600 × 1600 ≈ 40 亿条记录。

聚簇索引 vs 二级索引

聚簇索引（Clustered Index）：

  • 数据文件本身就是索引文件，叶子节点存储的是完整的行数据
  • InnoDB 中，主键就是聚簇索引
  • 如果没有显式定义主键，InnoDB 会：① 选第一个 UNIQUE NOT NULL 列；② 都没有则自动生成隐藏的 ROW_ID

二级索引（Secondary Index / 辅助索引）：

  • 叶子节点存储的是索引列的值 + 主键值
  • 通过二级索引查找数据需要回表：先查二级索引找到主键值，再用主键值去聚簇索引中查完整数据

回表示例：

执行过程：

  1. 在 name 的二级索引中找到 '张三' → id=1
  2. 拿着 id=1 回到聚簇索引中查找完整行数据
  3. 返回结果

⚠️ 回表是一次额外的 IO 操作，应尽量避免。

覆盖索引

覆盖索引：查询所需的所有列都包含在某个索引中，无需回表。

执行过程：

  1. 在 (name, age) 索引中找到 '张三' → age=20
  2. 所需数据（name, age）都在索引中，直接返回，不回表

💡 覆盖索引是重要的优化手段。设计索引时，考虑将常用查询的列包含进去。

联合索引与最左前缀原则

联合索引 (col1, col2, col3) 的排序规则：先按 col1 排序，col1 相同时按 col2 排序，col2 相同时按 col3 排序。

最左前缀原则：查询条件必须从联合索引的最左列开始，且不能跳过中间列，才能使用索引。

范围查询的影响：

  • a = 1：精确匹配，走索引 ✅
  • b > 10：范围查询，走索引 ✅
  • c = 5：b 已经是范围查询了，c 无法再使用索引进行查找 ❌（但如果是覆盖索引，c 的值仍可从索引中直接读取）

💡 设计建议：将等值查询的列放在联合索引前面，范围查询的列放后面。

事务与 ACID

什么是事务

事务是一组要么全部成功、要么全部失败的 SQL 操作，是数据库保证数据一致性的核心机制。

原子性（Atomicity）

含义：事务中的所有操作要么全部成功，要么全部失败回滚，不可分割。

底层实现——undo log（回滚日志）：

  • 事务开始前，先把修改前的旧值记录到 undo log
  • 如果事务失败或执行 ROLLBACK，通过 undo log 中的旧值进行恢复
  • undo log 本身也是持久化的，保证崩溃后也能回滚

一致性（Consistency）

含义：事务执行前后，数据库必须始终处于一致的状态（满足所有约束、触发器、级联回滚等规则）。

  • 不是由某个单一机制保证的，而是原子性 + 隔离性 + 持久性 + 约束检查共同作用的结果
  • 例如：外键约束、唯一约束、CHECK 约束等都在事务提交时检查

持久性（Durability）

含义：事务一旦提交，对数据的修改就是永久性的，即使数据库崩溃也不会丢失。

底层实现——redo log（重做日志）：

  • 事务执行时，先将修改操作记录到 redo log（顺序写入，速度快）
  • 然后才真正修改内存中的数据页（随机写，速度慢）
  • 事务提交时，确保 redo log 刷盘（fsync）
  • 崩溃恢复时，用 redo log 重放已提交但未写入磁盘的修改

💡 redo log 是顺序写（追加到文件末尾），比随机写磁盘快得多。这就是 WAL（Write-Ahead Logging，预写式日志）的核心思想：先写日志，再写数据。

隔离性（Isolation）

含义：多个事务并发执行时，一个事务的操作不应被其他事务干扰。

隔离级别（从低到高）：

隔离级别 脏读 不可重复读 幻读 实现方式
READ UNCOMMITTED ✅ 可能 ✅ 可能 ✅ 可能 基本不加锁
READ COMMITTED ❌ 不可能 ✅ 可能 ✅ 可能 每次读取都创建新 Read View
REPEATABLE READ（MySQL默认） ❌ 不可能 ❌ 不可能 ❌ 不可能（InnoDB 通过间隙锁解决） 事务开始时创建一次 Read View
SERIALIZABLE ❌ 不可能 ❌ 不可能 ❌ 不可能 所有读操作加共享锁

三种并发问题：

  • 脏读：事务 A 读到了事务 B 未提交的修改，B 回滚后 A 读到的是脏数据
  • 不可重复读：事务 A 在同一事务内两次读取同一行，结果不同（因为事务 B 在中间修改并提交了该行）
  • 幻读：事务 A 两次执行同一范围查询，返回的行数不同（因为事务 B 在中间插入了新行）

锁机制

行锁 vs 表锁

对比项 行锁 表锁
粒度 单行 整张表
并发度 高 低
开销 大（需要维护更多锁信息） 小
死锁概率 高 低
适用场景 OLTP（频繁增删改） OLAP（批量操作）

💡 InnoDB 的行锁是在索引上实现的。如果 WHERE 条件没有走索引，行锁会退化为表锁！

共享锁（S锁）与排他锁（X锁）

锁类型 简称 含义 兼容性
共享锁 S 锁 读锁，多个事务可同时持有 S 与 S 兼容，S 与 X 不兼容
排他锁 X 锁 写锁，只有一个事务能持有 X 与任何锁都不兼容

间隙锁（Gap Lock）

问题背景：行锁只能锁住已存在的行，无法阻止其他事务在”间隙”中插入新行，这会导致幻读。

间隙锁：锁住索引记录之间的间隙，防止其他事务在间隙中插入新记录。

⚠️ 间隙锁只在 REPEATABLE READ 隔离级别下生效。READ COMMITTED 下没有间隙锁。

临键锁（Next-Key Lock）

临键锁 = 行锁 + 间隙锁，锁住一个左开右闭的区间。

InnoDB 的默认行为：

  • 在 REPEATABLE READ 隔离级别下，所有 SELECT … FOR UPDATE 和 UPDATE/DELETE 语句都会使用临键锁
  • 这就是 InnoDB 能在 RR 级别下解决幻读问题的原因

MVCC（多版本并发控制）

MVCC 是 InnoDB 实现高并发非阻塞读的核心机制。它让读操作不需要加锁，同时还能保证事务隔离性。

核心组件

1. 隐藏字段

InnoDB 每行数据自动包含以下隐藏字段：

字段 说明
DB_TRX_ID 创建该行（或最后一次修改该行）的事务 ID
DB_ROLL_PTR 指向 undo log 中的回滚段，用于找到该行的历史版本
DB_ROW_ID 隐式行 ID（没有主键时使用）

2. undo log 版本链

每次修改一行数据时：

  • 旧版本被写入 undo log
  • 新版本行的 DB_ROLL_PTR 指向旧版本的 undo log 记录
  • 多个版本通过 DB_ROLL_PTR 串成一条版本链

3. Read View（读视图）

事务在读取数据时，会创建一个 Read View，包含：

字段 说明
m_ids 创建 Read View 时，活跃（未提交）的事务 ID 列表
min_trx_id m_ids 中的最小值
max_trx_id 创建 Read View 时，下一个将要分配的事务 ID
creator_trx_id 创建该 Read View 的事务 ID

MVCC 可见性判断算法

当事务要读取某一行时，从最新版本开始，沿着版本链逐个检查：

不同隔离级别下 Read View 的创建时机

READ COMMITTED：

  • 每次执行 SELECT 时都创建一个新的 Read View
  • 所以能读到其他事务已提交的最新修改（不可重复读）

REPEATABLE READ：

  • 事务中第一次执行 SELECT 时创建 Read View，之后复用
  • 所以整个事务期间看到的数据快照一致（可重复读）

MVCC 完整流程示例

💡 MVCC 只适用于快照读（普通 SELECT）。当前读（SELECT … FOR UPDATE、UPDATE、DELETE）会加锁，不走 MVCC。

📌 总结：各机制之间的关系

  • 索引决定了数据的组织方式和查询效率
  • 锁保证了并发写入时的数据一致性
  • MVCC 让读操作不加锁，大幅提升并发性能
  • undo log 同时服务于原子性（回滚）和 MVCC（多版本）
  • redo log 保证了持久性

这些机制相互配合，共同构成了 InnoDB 存储引擎的核心能力。

结构体内存对齐

发表于 2026-05-20 分类于 C_C++
本文字数： 3k 阅读时长 ≈ 4 分钟
本文详细介绍结构体内存对齐。

结构体内存对齐，简单说就是编译器在分配内存时，并不是简单地把结构体成员一个接一个紧挨着存放，而是会按照某种规则，在成员之间、甚至在结构体末尾插入一些空白字节（填充 padding），使得每个成员的地址都满足特定的对齐要求。

下面从三个层面来详细拆解：是什么、为什么要、不对齐会怎样。

一、结构体内存对齐是什么？

1. 自然对齐规则

每种数据类型都有一个“对齐值”（alignment），通常是它自身的大小，比如：

  • char / bool —— 1 字节对齐（可以放在任何地址）
  • short —— 2 字节对齐（地址必须是 2 的倍数）
  • int / float —— 4 字节对齐（地址必须是 4 的倍数）
  • double / long long —— 8 字节对齐（地址必须是 8 的倍数，在 64 位平台）
  • 指针 —— 8 字节对齐（64 位）

结构体内部成员排列时，必须保证每个成员的起始地址相对于结构体起始地址的偏移量，是这个成员自身对齐值的整数倍。

同时，为了保证结构体数组也能正确对齐，结构体的总大小必须是最大对齐成员对齐值的整数倍，必要时在末尾额外填充。

实际对齐值 = min(成员自身对齐值, 指定对齐值)。

例如 #pragma pack(1) 强制按 1 字节对齐，上面的 Demo 将变成 6 字节，没有任何填充。
C++11 起还提供了 alignof 查询对齐值、alignas 指定对齐。

二、为什么要对齐？

1. 硬件要求：CPU 访问内存的对齐限制

现代 CPU 并不是一个字节一个字节地读写内存，而是以“字（word）”、“双字”等为单位，通过数据总线与内存交互。
例如，一个 64 位 CPU 一次可以从内存读取 8 个对齐的字节。如果数据跨越了对齐边界，CPU 就必须做额外的工作。

  • 有些 CPU 严格要求对齐（如许多 RISC 处理器：旧版 ARM、MIPS、SPARC 等），访问未对齐的地址会直接触发硬件异常（bus error），程序崩溃。
  • x86 架构允许未对齐访问，但会付出性能代价：CPU 需要做两次内存访问，然后拼接字节，这比对齐访问慢很多。

2. 性能：减少内存访问次数

对齐保证了一个基本类型的数据不会“跨两次读取”：

  • 假设一个 4 字节 int 存放在地址 0x1001（未对齐）。CPU 要读取它，需要先读 0x1000～0x1003 这 4 个字节，再读 0x1004～0x1007，然后移位、拼接，才能得到完整数据。
  • 而对齐到 0x1000 或 0x1004 时，一次 4 字节读取就搞定。

对于频繁访问的结构体（如游戏循环中的向量、数据库缓冲区的记录），这个开销会被放大，直接影响吞吐量。

3. 原子操作的保证

多线程编程中，许多原子操作（如 std::atomic）要求操作对象自然对齐。
如果不对齐，读写可能不是原子的：一个线程在两次内存访问之间，另一个线程可能看到了“一半旧值一半新值”，造成难以调试的竞争。

4. 缓存行与伪共享

CPU 缓存以“缓存行”为单位（通常 64 字节）。
如果结构体成员没有对齐好，一个关键字段可能横跨两个缓存行，导致缓存命中率下降。
在多核编程中，还会因此引入“伪共享”（多个核修改同一缓存行的不同变量，互相使缓存失效），性能急剧降低。有时甚至需要用 alignas(64) 强制对齐到缓存行边界。

5. 标准规定与编译器优化

C/C++ 标准规定，通过未对齐的指针访问数据属于未定义行为（UB）。
这意味着编译器在生成代码时可以假设指针是对齐的，从而使用更高效的 SSE/AVX 等对齐加载指令。如果你偷偷传了一个未对齐地址，程序可能直接崩掉，或者得到错误结果。

三、不对齐会怎么样？

“不对齐”通常指的是通过某些手段强行紧凑存放结构体（如 __attribute__((packed))），导致成员地址不再满足自然对齐。这可能带来以下后果：

1. 程序崩溃

在不允许未对齐访问的硬件上（某些 ARM、MIPS、SPARC），执行 packed 结构体成员的读写，会立刻触发 SIGBUS 或类似的硬件异常。

即使是 x86，当使用 SIMD 指令（如 movaps 要求 16 字节对齐）去操作未对齐的数据时，也会引发一般保护错误。

2. 性能急剧下降

即使硬件支持未对齐访问（如 x86），未对齐读写比对齐读写慢好几倍。
编译器在知道某个指针可能未对齐时（如 packed 结构体成员的地址），往往被迫生成更保守的代码，比如用多次单字节移动代替一次 mov，或者使用专门处理未对齐的指令，拖累所有相关代码的性能。

3. 原子性丧失，多线程崩溃

一个未对齐的 int 在多核环境下，可能被拆成两次内存事务，导致读-修改-写操作不是原子的，进而导致计数器错误、状态不一致等灾难性问题。

4. 触发未定义行为

如果你把 packed 结构体中某个未对齐的 int 成员的地址取出来，强转成一个普通的 int*，然后解引用，这在 C/C++ 标准中就是 UB。
编译器优化时可能会假定这个 int* 是 4 字节对齐的，从而产生错误的代码，不会给你任何警告。

5. 代码不可移植

依赖未对齐的代码，可能在你的 x86 开发机上跑得好好的，部署到 ARM 服务器或嵌入式设备上就崩溃，排查起来极其痛苦。

四、总结

  • 结构体内存对齐是编译器自动完成的工作，遵循“每个成员放在其对齐倍数的偏移上，总大小是最大对齐倍数”的原则。
  • 为什么要对齐？ 为了硬件兼容、高性能、原子性保证、标准合规。本质上是一种以空间换时间的优化策略。
  • 不对齐会怎样？ 轻则性能骤降，重则崩溃、数据损坏、难以调试的并发 bug。

在日常开发中，除非在做网络协议解析、二进制文件格式映射、内存极度受限的嵌入式场景，一般不推荐盲目使用 packed。当你必须紧凑存放时，也请清楚其代价，并务必做好跨平台测试。如果只是为了节省内存，不妨先试试重排结构体成员顺序——它完全免费且安全。

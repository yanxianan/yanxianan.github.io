malloc和free

发表于 2026-05-20 分类于 C_C++
本文字数： 1.3k 阅读时长 ≈ 2 分钟
本文详细介绍malloc和free。

1. malloc 出来的内存在哪？

简单说：在堆（heap）上。

我们可以从程序的内存布局来理解。一个典型的 C 程序在虚拟内存中大致分为这几个区域：

  • 栈：存放局部变量、函数参数、返回地址等，由编译器自动管理。
  • 堆：由 malloc / calloc / realloc 等动态分配的内存块就在这里，需要程序员手动用 free 释放。
  • 数据段：全局变量、静态变量等。
  • 代码段：存放可执行指令。

当你调用 malloc(100) 时，它会从堆中找一块足够大的、未被使用的连续地址空间，返回这块空间的首地址。堆向上增长（通常），而栈向下增长。

2. free 之后指针要不要置 NULL？为什么？

强烈建议置 NULL，但这并不能解决所有问题，要先理解它的作用和局限。

如果不置 NULL，会有什么问题？

调用 free(p) 后，内存被归还给堆管理器，但指针 p 本身还保存着原来的地址。这时 p 就变成了一个悬垂指针，也就是指向已经释放内存的指针。

有两个典型危害：

  1. 误用（解引用悬垂指针）
    继续通过 p 读写数据，属于未定义行为。这块内存可能已被分配给其他变量，你可能会破坏他人数据，或者读到脏数据。

  2. 重复释放
    不小心对同一个地址再次调用 free 会导致 double free，这通常会破坏堆的内部结构，引发崩溃或安全漏洞。

置 NULL 的好处

只要执行 p = NULL;，就能避免上述两个问题：

  • 误用会立马暴露：解引用空指针通常会导致段错误，让错误立刻浮现，便于调试（比静默地破坏内存要好得多）。
  • 重复释放变安全：C 标准规定 free(NULL) 是安全的，什么都不做。所以多次 free(NULL) 不会引发 double free 问题。

置 NULL 的局限：指针别名

置 NULL 只能保护这一个指针变量。如果有多个指针指向同一块内存，只把其中一个置 NULL，其它的指针依然指向已释放的内存。

所以核心是要在逻辑上彻底结束所有对该内存的引用。置 NULL 能防止从某个变量误用，但对多指针情况需要更严格的生命周期管理。

实践建议

  1. free 后紧接着置 NULL，特别是对全局指针、结构体成员指针，或者函数返回后还存在的指针。
  2. 如果指针是函数的局部变量，且 free 后马上退出函数，不置 NULL 也问题不大（因为指针本身随栈帧销毁）。但为了统一防御习惯，很多团队规定一律置 NULL。
  3. 对于多指针情况，最好明确“谁持有所有权”来释放，其他指针只读并不再使用。

一句话总结：malloc 的内存来自堆；free 后把指针置 NULL 可以防误用、防重复释放，是一个低成本高收益的安全习惯。

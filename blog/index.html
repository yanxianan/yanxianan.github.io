【ZYNQ开发踩坑记录】程序JTAG调试正常，固化到Flash断电后不运行的排查与解决

在 ZYNQ 的嵌入式开发（Bare-metal 或 FreeRTOS）过程中，你是否遇到过这种“幽灵”现象：程序在 Vitis / SDK 中通过 JTAG 下载调试时一切完美，可一旦打包成 BOOT.bin 烧写到 QSPI Flash，断电重启后板子就像死机了一样毫无反应？但神奇的是，如果你此时用 Vitis 的 “Attach to running target” 连上去，程序又奇迹般地开始往下跑了？

如果你也遇到了这个症状，别怀疑芯片坏了，这大概率是 标准输出（stdout）配置 给你挖的坑。本文将详细复盘该问题的现象、根本原因及两种解决办法。

🐞 现象描述

  • JTAG 在线调试模式： 程序正常运行，逻辑完美。
  • QSPI Flash 固化启动模式： 烧写成功后，断电重启，程序毫无反应，LED 不闪，外设不工作。
  • 关键线索： 在 Flash 启动“死机”的状态下，打开 Vitis 的 Debug 功能，选择 Attach to running targets（仅连接不复位下载），连接上的瞬间，程序突然“活”了过来，开始正常执行。

🔍 根本原因分析：被 printf 与 coresight 堵死的 CPU

这个现象的罪魁祸首，通常是代码中使用了 printf 打印函数，并且 Board Support Package (BSP) 中的标准输入输出被配置为了半主机模式（Semihosting）。

深层逻辑如下：

  1. Vitis 的默认行为： 如果你在 Vivado 的 Block Design 中没有使能 ZYNQ PS 端的物理串口（UART0 或 UART1），Vitis 导出硬件后，会发现没有物理串口可用。为了让你依然能使用 printf 调试，BSP 会默认将 stdin 和 stdout 映射到一个叫 coresight 的虚拟通道上。
  2. JTAG 模式下的工作状态： 当通过 JTAG 调试时，coresight 通道畅通。处理器执行 printf 时，数据会通过 JTAG 传回电脑，在 Vitis 的控制台打印出来，一切看起来都很正常。
  3. Flash 脱机模式下的致命堵塞： 当板子断电重启，脱离 JTAG 独立运行时，物理上根本没有 JTAG 调试器在接收这些数据。当代码执行到 printf 时，底层缓冲区迅速填满，处理器会一直处于阻塞（挂起）状态，死等 JTAG 调试器来把数据读走。 此时，程序在宏观上表现为“死机”。
  4. 为什么 Attach 后又能跑了？ 当你使用 “Attach to running target” 连上板子时，JTAG 通道瞬间被打通，积压在缓冲区里的打印数据被读走，阻塞条件解除，CPU 被释放，程序自然就继续往下跑了。

🛠️ 解决办法

根据你的实际需求，这里提供两种解决方案。

方案一：不需要串口打印（快速验证法）

如果你的程序目前不需要在终端打印任何调试信息，只需要它能脱机跑起来，直接在 Vitis 中关闭标准输入输出即可。

操作步骤：

  1. 在 Vitis 的 Explorer 窗口中，双击打开你的 Platform 工程 中的 platform.spr 文件（或者直接打开 BSP 配置界面 Board Support Package Settings）。
  2. 在左侧导航栏找到 Overview -> standalone。
  3. 在右侧的配置列表中，找到 stdin 和 stdout。
  4. 点击下拉菜单，将它们的值从 coresight 修改为 none。
  5. 点击 OK 保存。
  6. 右键点击你的平台工程和 APP 工程，依次选择 Clean Project 和 Build Project。
  7. 重新打包生成 BOOT.bin，烧写到 Flash。断电重启，问题解决！

注意： 此时代码中的 printf 实际上变成了空操作，不会引发阻塞，但你也无法看到任何打印信息。

方案二：需要串口打印（终极规范做法）

如果你后续开发必须依赖串口打印调试信息，就需要回到硬件源头，把真实的硬件串口打开。

操作步骤：

  1. 回到 Vivado： 打开 Block Design，双击 ZYNQ7000 处理器核。
  2. 使能串口： 在 MIO Configuration (或 Peripheral I/O Pins) 中，根据你开发板的原理图（通常是 UART1 对应板载的 USB-TTL 芯片），勾选并使能 UART 0 或 UART 1。
  3. 重新生成硬件： 点击 OK，重新 Generate Output Products 和 Generate Bitstream。
  4. 导出硬件： File -> Export -> Export Hardware，覆盖旧的 .xsa 文件。
  5. 更新 Vitis 平台： 回到 Vitis，右键 Platform 工程，选择 Update Hardware Specification，导入最新的 .xsa 文件。
  6. 修改 BSP 设置： 再次打开 BSP 的 standalone 设置，此时 stdin 和 stdout 的下拉菜单中就会出现真实的物理串口（例如 ps7_uart_1）。选中它。
  7. 重新编译打包： Clean 并 Build 整个工程，重新生成 BOOT.bin。

这才是最规范的做法。烧写后，无论是 JTAG 调试还是 Flash 脱机运行，printf 都会将数据发往真实的物理串口，再也不会出现挂死的问题了。

📝 总结建议：

在进行 ZYNQ 裸机或 RTOS 开发时，如果遇到莫名其妙的卡死问题，不妨善用 Vitis 的 Attach to running target 加上 Pause (暂停) 功能。看看 Call Stack 停在什么函数里，往往能一针见血地找到真正的病因所在！

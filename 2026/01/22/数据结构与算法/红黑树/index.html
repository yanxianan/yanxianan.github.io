红黑树 (Red-Black Tree)

发表于 2026-01-22 分类于 数据结构与算法
本文字数： 1.4k 阅读时长 ≈ 2 分钟
本文介绍红黑树。

红黑树 (Red-Black Tree) 是计算机科学中最重要的数据结构之一，也是 C++ STL 中 map、set、multimap、multiset 的幕后功臣。

简而言之，它是一种自平衡的二叉搜索树 (Self-balancing Binary Search Tree)。

1. 为什么要发明红黑树？

为了解决普通二叉搜索树 (BST) 的退化问题。

  • 理想情况：BST 是平衡的，查找、插入效率是 $O(\log n)$。
  • 最坏情况：如果你按顺序插入数据（例如 1, 2, 3, 4, 5），普通的 BST 会退化成一条链表。这时候查找效率直接掉到 $O(n)$，非常慢。

红黑树通过制定一套“颜色规则”，强制树保持“大致平衡”，从而保证任何情况下的操作效率都是稳定的 $O(\log n)$。

2. 红黑树的 5 条铁律

红黑树的每个节点上多存了一个位 (bit) 的信息：颜色（红或黑）。

正是这几条看似奇怪的规则，保证了树的平衡：

  1. 节点颜色：每个节点要么是红色，要么是黑色。
  2. 根节点：根节点必须是黑色。
  3. 叶子节点：所有的叶子节点（NIL 节点，即空节点）都视为黑色。
  4. 红节点限制：不能有两个红节点相连（即红节点的父节点和子节点必须是黑色）。
  5. 黑色高度：从任意一个节点到其所有后代叶子节点的简单路径上，必须包含相同数量的黑色节点。

3. 为什么这能保证平衡？（核心直觉）

你可能会问：“这几条规则怎么就能保证树不变成链表呢？”

请结合规则 4 和规则 5 来看：

  • 规则 5 保证了每条路径上的黑色节点数量是一样的（我们把这看作树的“骨架”）。
  • 规则 4 限制了红色节点不能连在一起，这意味着红色节点只能穿插在黑色节点之间。

结论： 哪怕是一条红黑相间的最长路径，其长度也不会超过全黑的最短路径的 2 倍。

  • 最短路径：全黑。
  • 最长路径：黑-红-黑-红-黑… (红节点最多只能和黑节点一样多)。

这虽然不如 AVL 树（绝对平衡）那么严格，但足以保证效率在 $O(\log n)$ 级别。

4. 它是如何“自平衡”的？

当你插入或删除数据时，可能会破坏上面的规则（比如插入了一个红节点，结果父节点也是红的，违反规则 4）。

红黑树通过以下三种手段自我修复：

  1. 变色 (Recoloring)：把红变黑，或把黑变红。
  2. 左旋 (Left Rotation)：将某个节点向左“跌倒”，变成其右孩子的左孩子。
  3. 右旋 (Right Rotation)：将某个节点向右“跌倒”，变成其左孩子的右孩子。

比喻：想象你在玩魔方。当你改变了一块的位置（插入数据），魔方乱了（规则打破了）。你需要通过一系列特定的旋转和变色公式，让它重新符合规则。

5. 面试常考：红黑树 vs AVL 树

这是面试中最高频的对比问题。

特性 AVL 树 红黑树 (RB Tree)
平衡程度 严格平衡 (高度差不超过 1) 非严格平衡 (最长路径 <= 2倍最短路径)
查找效率 更快 (树更矮) 稍慢 (树稍高，但也属于 $O(\log n)$)
插入/删除 慢 (需要频繁旋转来维持严格平衡) 快 (旋转次数少，变色多)
应用场景 读多写少 (如字典库) 综合性能强，读写兼顾 (如 C++ STL map, Java TreeMap)

为什么 C++ STL 选择了红黑树？

因为通用库不知道用户是读得多还是写得多。红黑树在插入、删除和查找之间取得了最好的折中 (Trade-off)。它的统计性能在大多数场景下都优于 AVL 树。

总结

  1. 红黑树是一种近似平衡的二叉搜索树。
  2. 它通过颜色约束确保最长路径不超过最短路径的 2 倍。
  3. 它是 C++ STL map 和 set 的底层实现，保证了 $O(\log n)$ 的增删查改效率。

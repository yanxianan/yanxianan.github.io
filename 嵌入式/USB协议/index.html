USB协议

发表于 2026-04-28 分类于 嵌入式
本文字数： 3.8k 阅读时长 ≈ 5 分钟
本文详细介绍USB通信协议。

USB协议是过去二十多年里最成功、最普及的接口标准。它的能力远不止数据传输，其供电规范也已发展为一套完整的电源管理系统。下面我将为你拆解它的核心内涵。

🧱 一、协议架构：层次分明的系统工程

USB协议之所以能变得如此强大而稳定，得益于其清晰的分层架构设计。这套分层模型将复杂的通信任务分解，使得每一层可以独立发展演化，大幅降低了开发和维护的复杂度：

  • 物理层 (Physical Layer)：在USB 2.0时代，使用一对差分信号线D+/D-进行传输，抗干扰能力强。USB 3.0及以上则引入了额外的超高速差分对用于高速数据传输。

  • 数据链路层 (Link Layer)：确保数据点对点传输的可靠性。负责数据编码（NRZI）、差错检测（CRC校验） 以及对数据包进行确认（ACK） 或重传（NAK） 等控制。

  • 协议层 (Protocol Layer)：定义了底层的通信规则，是所有数据传输、设备枚举和端点配置等核心功能的基础。

  • 应用层 (Application Layer)：针对不同类型USB设备（如键盘鼠标、U盘、声卡）定义的特定功能规范，使得操作系统能加载对应的“客户端软件”（设备驱动），为用户提供具体功能。

🏗️ 二、数据流模型：从拓扑到行为

1. 物理拓扑：树状结构

在物理连接上，USB采用树状拓扑。主机（Host） 是树的根，其内部集成的根集线器（Root Hub） 提供初始端口。设备可直接连到根集线器，或通过外部集线器（Hub） 层层扩展，形成一个最多7层（包括根集线器）的结构。一个USB主机理论上最多可连接127个设备。

2. 行为模式：以主机为中心

USB的通信行为严格遵循主机为中心（Host-Centric） 的模式。这意味着，任何时候，必须是主机主动发起请求，设备才能被动响应。设备就像一个等待提问的职员，没有主机的允许，它不能主动“举手发言”。通过这种机制，主机可以精确地轮询和调度总线上的每一个设备，避免了数据冲突。

🔌 三、描述符与枚举：设备的“身份自白”

为了进行通讯，USB设备必须先向主机证明“我是谁”以及“我能做什么”。这个过程称为枚举，而设备是通过一套被称为描述符的数据结构来介绍自己的。就像一份递交给主机的详细简历：

  1. 设备描述符 (Device Descriptor)：简历的“基本信息”页。包含USB协议版本、厂商标识（VID）、产品标识（PID） 和可能的序列号等全局信息。一个设备只能有一份。
  2. 配置描述符 (Configuration Descriptor)：简历的“工作模式”页。一个设备可以有多种配置（如低功耗/高性能模式），主机检查后激活其中一种。它包含接口数量、最大功耗等信息。
  3. 接口描述符 (Interface Descriptor)：简历的“技能专长”页。一个配置可拥有多个接口，每个接口代表一个独立功能。例如，一个USB电话设备可能同时包含一个音频接口和一个HID接口。
  4. 端点描述符 (Endpoint Descriptor)：简历的“具体能力”页。每个接口有若干个端点（Endpoint），这是数据传输的终点和起点。端点描述符定义了传输方向（IN/OUT）、传输类型（控制/批量/中断/等时）和最大数据包长度等关键信息。

枚举过程，就是设备插入后，主机按照上述结构，自上而下、依次获取并解析这些描述符，最终加载合适驱动程序并激活设备的过程。

💧 四、端点、管道与关键组件

  • 端点 (Endpoint)：设备端的一个硬件“数据收发器”。它像一个有编号的房间（Endpoint Number），是数据的最终目的地或源发地。
  • 管道 (Pipe)：主机软件与设备端点之间的逻辑“通道”。建立了管道后，数据就能像水一样在其中定向流动。
  • 关键组件协作：
    • 主机端 (Host)：由客户端软件和负责硬件通信调度的主机控制器（Host Controller） 构成。
    • 设备端 (Device)：即物理硬件，是功能的最终执行者。
    • 通信分工：USB系统软件处理描述符获取、地址分配等通用任务，而客户端软件聚焦于设备特定功能，两者协作实现完整通信。

💬 五、数据包的微观世界

在总线上，所有操作都以**包（Packet）**为基本单位。虽然类型多样，但几乎所有USB包都遵循一个通用结构：“SYNC同步域 + PID包标识符 + 特定信息 + CRC校验 + EOP结束标识”。根据PID的不同，包又分为几大角色：

  • 令牌包 (Token Packet)：“发号施令者”，始终由主机发出。常见类型有IN（请求设备发送数据）、OUT（通知设备即将接收数据）、SETUP（启动控制传输）和SOF（帧起始，用于同步）。
  • 数据包 (Data Packet)：“关键信息载体”。主要有DATA0和DATA1，两者通过交替使用实现同步。例如，发送方发一个DATA0包，必须收到接收方ACK后，才发下一个DATA1包，以此判断对方是否收到。
  • 握手包 (Handshake Packet)：“状态报告员”。由数据接收方发出，告知发送方结果，如ACK（成功）、NAK（忙，请重试）、STALL（出错，需主机干预）。
  • 特殊包 (Special Packet)：用于特定用途，如集线器使用PRE包与低速设备通信。

🛠️ 六、从包到事务：一次完整交互

若干类型的包，按固定顺序组合起来，就完成了一次有意义的事务（Transaction）。典型的事务有：

  • SETUP事务：由令牌包(SETUP) + 数据包(DATA0) + 握手包(ACK) 组成，用于控制传输。
  • IN/OUT事务：对于IN（设备到主机），是令牌包(IN) → 数据包(DATA0/DATA1) → 握手包(ACK)。对于OUT（主机到设备），流程类似。

🧭 七、四大金刚：四种核心传输类型

为了解决不同类型数据的传输需求，USB定义了四种传输类型，它们是USB灵活性的基石。

传输类型 特点 应用场景 错误处理
控制传输 (Control) 非周期性、可靠、双向。用于命令/状态信息。所有设备必须支持。 设备枚举和配置过程，是设备的“管理通道”。 有完善的重试和错误报告机制。
批量传输 (Bulk) 非周期性、可靠、单向、大数据量。利用总线空闲带宽，无延迟保证。 U盘、打印机、扫描仪。 通过CRC校验保证数据100%正确，出错会重传。
中断传输 (Interrupt) 周期性、可靠、小数据量。主机保证在指定时间间隔内发起传输。 键盘、鼠标、游戏手柄。 有CRC校验和重传机制。
等时传输 (Isochronous) 周期性、不可靠、大数据量。保证固定的带宽和延迟，但不进行错误重传。 网络摄像头、USB声卡、麦克风。 无错误重传机制，宁可丢包也不延迟，以保证实时性。

⚡️ 八、不止于数据：统一供电的野心

USB从一开始就能提供5V电源，但功率有限。为了给笔记本等大功率设备充电，USB-IF制定了USB Power Delivery（USB PD） 规范，目前主要基于Type-C接口实现。

  • 技术核心：使用Type-C接口的CC（Configuration Channel） 引脚进行供受电双方的通信。
  • 版本与功率：
    • USB PD 2.0/3.0：支持最高100W (20V/5A) 的功率传输。
    • USB PD 3.1：引入扩展功率范围（EPR），将功率上限提升至240W (48V/5A)，足以驱动高性能笔记本。
    • USB PD 3.2：引入了更精细的电压调节机制（AVS），提升充电效率。
  • 充电模式：除了固定电压档位（5V, 9V, 15V, 20V等），还有PPS（可编程电源） 模式，允许设备以毫伏级步进实时调整电压/电流，实现更精准高效的充电。

🚀 九、演进之路：从 1.5Mbps 到 40Gbps

版本 最大速率 发布年份 核心特点与意义
USB 1.0/1.1 1.5 Mbps (低速)
12 Mbps (全速)		 1996/1998 确立了即插即用，引入HUB拓扑。
USB 2.0 480 Mbps (高速) 2000 普及的关键版本，至今许多MCU仍在使用。引入OTG、微帧机制。
USB 3.2 Gen 1 5 Gbps 2008 开启超高速时代，引入全双工通信，接口新增5个引脚。
USB 3.2 Gen 2 10 Gbps 2013 提升速率，引入128b/132b高效编码。
USB 3.2 Gen 2x2 20 Gbps 2017 利用Type-C双通道设计，速率翻倍；完全使用Type-C接口。
USB4 20/40 Gbps 2019 融合雷电3协议，动态带宽分配，可同时承载数据、视频、充电。

💎 总结

至此，USB协议的完整图景已经清晰。它以一套分层的体系结构为骨架，通过精妙的数据流模型（以主机为中心）和自描述机制（描述符与枚举）组织通信，并凭借包、事务、传输的三级传递体系，以及四种核心传输类型，灵活应对各种应用场景。在此基础上，USB PD供电规范赋予了它强大的能量，而版本的持续演进则让它始终立于技术潮头。

USB的成功，根植于它对复杂性出色的分层封装：底层的物理信号、数据包格式等问题对普通用户完全透明，开发者也可以根据需求在不同协议层工作。这也与RS-232等只关注物理层的简单串口标准形成了鲜明对比。

如果你对其中某个细节，比如枚举的数据包交互、PD充电的协商过程，或者USB4与雷电3的关系等有进一步兴趣，我们可以继续深入探讨。

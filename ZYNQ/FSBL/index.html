本文介绍ZYNQ裸机上电的第二个阶段FSBL。

  • ️ ZYNQ启动流程第二阶段：FSBL详解

    FSBL（First Stage Boot Loader）是ZYNQ启动流程中首个由用户可定制的软件阶段。它在BootROM完成基础硬件唤醒后立即执行，若FSBL执行失败，系统将无法进入后续阶段。

    其核心使命概括为：完成PS端关键外设初始化、配置PL逻辑、加载并移交控制权至下一阶段程序（SSBL或裸机应用）。

    运行位置：片上内存（OCM）

    • 物理位置：ZYNQ芯片内部的 256KB SRAM。
    • 地址映射：通常位于 0xFFFF_0000 至 0xFFFFFFFF 区域。
    • 可用空间限制：
      • BootROM阶段会占用 高64KB（用于存储启动参数和状态）。
      • FSBL实际可用空间 ≤192KB（低192KB区域）。
      • 注意：若FSBL代码超过192KB，需改用 XIP（就地执行）模式直接从Flash运行，但这会导致启动速度下降。

    ️ 一、FSBL的核心作用

    1. 硬件初始化（关键基础）

    • DDR控制器初始化：执行完整的DDR PHY训练流程（包括时序校准、信号完整性测试）。这是FSBL最核心且最易出错的环节，若失败（如时序参数与硬件不匹配），后续所有内存操作均会崩溃。
    • PS端外设配置：根据Vivado生成的硬件描述文件（HDF/XSA），初始化MIO引脚、时钟树（PLL）、UART（用于调试输出）、GPIO等基础外设。

  • 关键限制：FSBL运行于无MMU的裸机环境，代码必须严格控制在192KB以内，超限将导致启动异常。

2. PL逻辑配置（可选但关键）

  • 比特流加载：通过PCAP（Processor Configuration Access Port）接口，将PL比特流文件（.bit）从启动设备加载至FPGA逻辑单元。
  • 时序严格性：需精确控制 PROG_B 复位脉冲（≥300ns）、INIT_B 响应检测（超时阈值建议1ms），时序偏差会导致PL配置失败。
  • 非必选项：若系统仅使用PS端（如纯ARM裸机应用），可跳过PL配置步骤。

3. 下一阶段程序加载

  • 目标程序类型：

    • SSBL（如U-Boot）：用于Linux系统启动。
    • 裸机应用程序：直接运行无操作系统的用户代码（如 helloworld.elf）。
    • 加载机制：FSBL解析BOOT.BIN中的分区表，将目标程序从启动设备复制到DDR指定地址，并在跳转前禁用MMU（因Linux内核要求启动时MMU处于关闭状态）。

    二、FSBL的详细执行流程

    1. 启动入口与环境准备

  • 执行起点：BootROM将FSBL加载至OCM（通常地址 0x00000000），并跳转至其入口点（_vector_table）。

    • 初始环境：CPU运行于SVC特权模式；MMU与缓存（Cache）处于关闭状态；仅依赖OCM作为临时存储空间。

    2. 关键初始化步骤

    • 硬件抽象层（HAL）初始化：解析HDF/XSA文件中的硬件描述信息，生成外设寄存器映射表。HDF文件是FSBL的“配置中枢”，若其中 clock-frequency 等参数错误，将直接导致UART波特率计算偏差，引发通信乱码。
    • DDR内存初始化：包含三阶段训练流程（DQS训练校准数据选通信号时序、写入训练优化数据写入窗口、读取训练校准数据读取窗口）。DDR初始化失败是裸机开发中最常见的启动卡死原因，通常因Vivado中DDR配置与实际硬件（如芯片型号、PCB走线）不匹配所致。

  • PL比特流加载（若启用）：通过PCAP接口通信，利用DMA将比特流数据从启动设备传输至PL（传输长度必须4字节对齐，否则触发硬件异常）。期间需持续轮询PCAP状态寄存器，确认 DMA_DONE 标志置位后才继续后续流程。

    • 下一阶段程序加载：遍历BOOT.BIN中的分区头表，定位SSBL或裸机程序的偏移地址与大小。若启用安全启动，FSBL会验证下一阶段镜像的数字签名（如RSA校验），跳过此步将导致系统易受固件篡改攻击。

  • 控制权移交：执行清空BSS段（未初始化全局变量置零）、设置堆栈指针（SP）至DDR中的有效地址，最后跳转至目标程序入口地址（通过函数指针调用）。移交前MMU必须保持禁用，否则Linux内核将无法正确初始化虚拟内存。

    三、FSBL开发与调试要点

    1. 生成与定制方法

    • 标准生成流程：在Vivado导出硬件平台（.xsa）后，于Vitis中创建应用工程 → 选择模板 Zynq FSBL → 自动生成源码（fsbl_main.c 等）。

  • 关键可定制点：修改 fsbl_hooks.c 中的钩子函数（如 FsblHookBeforeHandoff()），插入自定义逻辑；调整HDF中的时钟/DDR参数以适配硬件。

    2. 常见故障与排查

    • 启动卡死在FSBL阶段：
    • 典型现象：串口无输出或输出中断于“DDR init”环节。
      • 根因与解决：检查Vivado中DDR PHY参数是否与开发板手册一致（如 MT41K256M16 需匹配时序表）；精简FSBL代码（如关闭 FSBL_DEBUG_INFO 宏减少日志输出）以解决OCM超限问题。

  • PL配置失败：

    • 典型现象：PL端逻辑功能异常，或串口输出 Bitstream load failed。
    • 根因与解决：确认Create Boot Image时已添加 .bit 文件；在FSBL源码中调整 XFsbl_PcapWaitForDmaDone() 的超时阈值以解决PCAP时序超限。

    3. 调试核心手段

    • 串口日志：启用 FSBL_DEBUG_INFO 宏（在 fsbl_debug.h 中定义），通过UART输出各阶段状态（如DDR训练结果）。

  • JTAG断点：在Vitis中设置断点于 main() 函数，单步跟踪初始化流程。

    • 寄存器快照：使用XSCT脚本读取关键寄存器（如DDR控制器状态寄存器 0xF8006000）。

    四、FSBL与裸机开发的关联

  • 裸机程序的启动依赖：裸机应用（如 helloworld.elf）必须通过FSBL加载至DDR，无法直接由BootROM启动（因BootROM不支持DDR初始化）。

    • FSBL的裁剪必要性：纯裸机场景可删除FSBL中SSBL/U-Boot相关代码，仅保留PS初始化与应用程序加载逻辑，以缩短启动时间。

  • 关键区别：若跳过FSBL（如JTAG直接加载裸机程序），PL端逻辑将无法自动配置，需手动通过JTAG下载比特流。

总结：FSBL是ZYNQ启动链的关键信任锚点，其稳定性直接决定系统能否进入用户代码阶段。开发裸机应用时，必须确保FSBL正确完成DDR初始化与PL配置（若需要），否则后续程序必然崩溃。建议首次开发时优先验证FSBL日志，确认“DDR init done”及“Handoff to application”等关键输出后再调试应用逻辑。

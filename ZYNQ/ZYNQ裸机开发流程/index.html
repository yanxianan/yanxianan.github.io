ZYNQ裸机开发流程

发表于 2026-05-06 更新于 2026-05-11 分类于 ZYNQ
本文字数： 2.8k 阅读时长 ≈ 4 分钟
本文介绍ZYNQ裸机开发的流程及基础知识。

ZYNQ裸机开发是指直接在ARM处理器（PS端）上运行不依赖操作系统的底层代码，通过寄存器操作控制硬件资源（包括PL端逻辑）的开发方式。其核心流程为：使用Vivado完成硬件设计并导出平台文件 → 在SDK/Vitis中创建裸机应用工程 → 编译生成包含FSBL、比特流和应用程序的BOOT.BIN镜像 → 烧录至开发板启动。以下结合关键步骤与原理进行详细说明：

一、基础概念与开发环境

1. ZYNQ架构核心组成

  • PS端（Processing System）：集成ARM Cortex-A9（Zynq-7000）或Cortex-A53/R5（Zynq UltraScale+）处理器，负责运行软件代码，裸机开发即直接操作PS端外设寄存器。
  • PL端（Programmable Logic）：FPGA逻辑部分，通过AXI总线与PS端交互，需在Vivado中设计并生成比特流文件（.bit）。
  • 裸机（Baremetal）定义：不依赖操作系统，直接调用Xilinx提供的硬件抽象层（HAL）库函数操作寄存器，适用于实时性要求高、资源受限的场景。

2. 开发工具链

  • Vivado：用于设计PL端逻辑、配置PS端外设（如DDR、UART）、生成硬件平台文件（.xsa或.hdf）。

  • SDK（2018.3及以前）或Vitis（2019.1+）：基于Eclipse的IDE，用于编写裸机代码、编译生成可执行文件（.elf）及制作启动镜像。

    • 关键区别：Vitis是SDK的升级版，但裸机开发流程基本一致，仅界面和部分操作路径略有差异。

二、裸机开发详细流程

1. 硬件设计（Vivado阶段）

（1）创建工程与添加ZYNQ IP

  • 新建RTL工程，选择目标器件（如xc7z020clg400-1）。

  • 在Block Design中添加 ZYNQ7 Processing System（Zynq-7000）或ZYNQ UltraScale+ MPSoC（UltraScale+）IP核。

  • 关键配置项：

    • Clock Configuration：设置输入时钟频率（通常50MHz）及输出频率（如CPU主频667MHz、DDR 533MHz）。
    • DDR Configuration：必须与开发板实际DDR型号匹配（如MT41K256M16），否则FSBL初始化失败。
    • MIO/EMIO配置：启用UART（调试输出）、GPIO等必要外设，电压标准需与硬件一致（如3.3V）。

（2）生成比特流与导出硬件平台

  • 自动连接PS-PL接口（AXI总线），运行综合、实现，生成比特流文件（.bit）。

  • 导出硬件平台文件：

    • Vivado 2018.3及以前：导出.hdf文件（File → Export → Export Hardware）。

  • Vivado 2019.1+：导出.xsa文件（Vitis仅支持.xsa）。

三、软件开发（SDK/Vitis阶段）

1. 创建裸机应用工程

  • 启动SDK/Vitis：在Vivado中点击File → Launch SDK/Vitis，或直接打开工具并导入.xsa/.hdf文件。

  • 新建应用工程：

    • 选择File → New → Application Project。

    • 关键参数：

      • OS Platform：必须选择standalone（裸机模式）。
      • Processor：选择PS端对应ARM核（如ps7_cortexa9_0）。
      • Templates：初学者选Hello World模板，自动生成基础代码框架。

2. 编写与编译代码

  • 核心文件：

    • helloworld.c：主程序入口，包含main()函数。
    • platform.h：由BSP自动生成，包含硬件初始化函数（如init_platform()）。

  • 关键操作：

    • 直接调用HAL库函数：例如通过XUartPs_Send()操作UART，避免使用Linux系统调用。
    • 硬件抽象层（HAL）作用：将寄存器操作封装为C函数（如Xil_Out32(addr, value)），屏蔽底层硬件差异。

  • 编译工程：点击工具栏“锤子”图标生成.elf文件。

四、启动流程与镜像制作

1. ZYNQ启动四阶段

  1. Stage 0：BootROM
    芯片上电后，从固化ROM执行代码，根据MIO引脚状态选择启动设备（如SD卡、QSPI Flash）。
  2. Stage 1：FSBL（First Stage Boot Loader）
    • 初始化DDR控制器（关键步骤，失败则无法继续）。
    • 配置PL端逻辑（通过PCAP接口加载比特流）。
    • 加载应用程序至DDR，跳转至main()入口。
  3. Stage 2：裸机应用程序
    直接执行用户代码（无操作系统介入）。
  4. （可选）Stage 3：SSBL
    裸机开发中通常跳过此阶段（仅Linux开发需U-Boot）。

2. 制作BOOT.BIN启动镜像

  • 必要组件（严格顺序）：

    1. FSBL.elf：由SDK/Vitis自动生成（File → New → Application Project → 选择Zynq FSBL模板）。
    2. 比特流文件（.bit）：PL端配置数据。
    3. 应用程序.elf：用户编译生成的裸机代码。

  • 操作步骤：

    • 在SDK/Vitis中选择Xilinx → Create Boot Image。
    • 添加上述三个文件，指定输出路径（生成BOOT.BIN）。
    • 关键验证：确保FSBL在首位，否则PL无法初始化。

3. 烧录与启动

  • 将BOOT.BIN复制到SD卡根目录（FAT32格式），或烧录至QSPI Flash。
  • 设置启动模式：根据开发板原理图，配置MIO[8:2]引脚为SD卡/QSPI启动模式（通常通过拨码开关）。
  • 上电后，BootROM自动加载BOOT.BIN，依次执行FSBL→应用程序。

五、关键注意事项

1. 常见问题排查

  • DDR初始化失败：
    检查Vivado中DDR配置是否与开发板实际芯片型号完全一致（如时序参数、电压），这是FSBL阶段最常见错误。
  • PL未配置成功：
    确认比特流文件已包含在BOOT.BIN中，且FSBL版本与硬件设计匹配（不同Vivado版本生成的FSBL可能不兼容）。
  • 串口无输出：
    检查MIO UART引脚配置、波特率设置（默认115200），以及代码中是否调用init_uart()初始化。

2. 裸机开发核心优势

  • 实时性高：无操作系统调度开销，中断响应延迟可低至微秒级。
  • 资源占用小：代码直接运行在DDR/OCM中，无需Linux内核的数百MB内存。
  • 硬件控制精细：可直接操作寄存器，适合驱动开发、实时控制等场景。
